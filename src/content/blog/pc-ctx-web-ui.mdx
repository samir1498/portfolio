My AI agents can read and write plans via CLI and MCP. But I can't. I don't want to open a terminal to check what's active, what's blocked, or how this week compares to last week.

So I built a web UI.

## The context problem

If you work with AI coding agents like OpenCode, Claude Code, Cline, Cursor: you've seen the pattern:

- CLAUDE.md files with rules
- Plans stored as markdown
- "Read the plans before you start" prompts

It works, but it scales poorly. There's no overview. No way to see what's active across all projects. No historical trend.

pc-ctx started as a CLI to manage plans deterministically: `ctx plan add`, `ctx status`, `ctx validate`. Then an MCP server so agents could call tools directly. The CLI is great for agents. The web UI is great for humans. Same data, two interfaces.

## Why not a database?

The whole system is markdown files in a git repo. YAML frontmatter. That's it. No database, no ORM, no API server to maintain.

For the web UI, I needed a read-only view into that repo. The GitHub API is already there, authenticated, serving live data. I just needed to proxy it through a Cloudflare Worker.

Result: zero infrastructure. The Hono Worker deploys with `wrangler pages deploy`, reads from GitHub API with a PAT, and serves the React SPA. To scale, I do nothing. Cloudflare handles it.

## Architecture

```
Browser → Cloudflare Access → Hono Worker → GitHub API → personal-context repo
                     ↓
              React SPA
         (served by Worker)
```

The Worker proxies GitHub API calls and serves the SPA. Cloudflare Access handles auth at the edge. No auth code, no sessions, no JWTs.

## What the dashboard shows

![Dashboard overview](/blog/pc-ctx-web-ui/dashboard.png)

### Stat cards

Four cards at the top showing counts for Plans, Roadmaps, References, and Progress logs. Numbers count up from zero on scroll.

### Status bar

A horizontal bar showing active / paused / done / cancelled distribution across all plans. Color-coded segments at a glance.

### Weekly activity bars

Bars showing how many task entries were logged per week. Data comes from progress files that follow a `2026-W25.md` naming pattern.

### Top plans + Recent timeline

Priority-sorted plans with recent progress logs as a timeline. The two sections give you "what's next" and "what just happened" in one view.

## The hardest part (and the fix)

The GitHub Contents API returns a `download_url` for each file pointing to `raw.githubusercontent.com`. For public repos, this works. For private repos, `raw.githubusercontent.com` **ignores Bearer auth headers**, so the file body was always null.

Fix: use `Accept: application/vnd.github.raw+json` on `api.github.com/contents/...` instead. This returns the raw file content directly, with proper auth handling.

```ts
const res = await fetch(url, {
  headers: {
    Authorization: `Bearer ${token}`,
    Accept: "application/vnd.github.raw+json",
  },
});
```

## Cloudflare Access

The entire site is private by default. Cloudflare Access sits at the edge and checks auth before traffic reaches the Worker. I configured it in the Zero Trust dashboard: select "Workers" as the app type, pick `personal-context.pages.dev`, set a policy rule.

No auth code. No sessions. No JWTs. It just works.
