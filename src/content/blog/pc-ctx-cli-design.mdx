Post one was the vision. A folder of markdown files with MCP tools so agents don't read files. This one is the CLI itself. The bridge between "agent without MCP" and "structured data on disk."

The code is at [github.com/samir1498/pc-ctx](https://github.com/samir1498/pc-ctx), package `@pc-ctx/cli`.

## Why citty

citty is from the unjs ecosystem. Works with Bun, clean API, no classes. You pass it a config object with `meta`, `args`, and `run`.

```typescript
import { defineCommand, createMain } from "citty";

const listCmd = defineCommand({
  meta: { name: "list", description: "List all plans" },
  args: {
    status: {
      type: "string",
      description: "Filter by status",
      required: false,
    },
  },
  run({ args }) {
    // args.status: typed string or undefined
  },
});
```

Compose with `subCommands`:

```typescript
const mainCmd = defineCommand({
  meta: { name: "ctx" },
  subCommands: { list: listCmd, show: showCmd, status: statusCmd },
});
const main = createMain(mainCmd);
```

citty handles help generation, arg parsing, routing. I write business logic.

## The factory pattern

Every domain follows the same cycle. List files, parse YAML, display results. Plans, roadmaps, research, ideas, processes, progress, references, archive. All the same shape.

I wrote one function:

```typescript
function makeDomainCmd(name, description, dir) {
  const listCmd = defineCommand({
    /* list files in dir */
  });
  const showCmd = defineCommand({
    /* show file by slug */
  });
  const addCmd = defineCommand({
    /* create file with frontmatter */
  });
  return defineCommand({
    meta: { name, description },
    subCommands: { list, show, add },
  });
}

const ideasCmd = makeDomainCmd("ideas", "Manage ideas", IDEAS_DIR);
const processesCmd = makeDomainCmd(
  "processes",
  "Manage process docs",
  PROCESSES_DIR,
);
```

One function, seven domains, twenty-one commands. Add a new domain tomorrow and it's three lines.

## Validation is the API

Agents parse CLI output when they can't use MCP. If the output is malformed, the agent breaks. If frontmatter has a typo in a status field, the agent reads an invalid plan and either crashes or ignores it.

Validation isn't an afterthought. It runs before every write. Per-domain rules let plans be strict and ideas be loose, all in the same loop.

More importantly, it runs in CI. Bad data doesn't merge.

## The core split

```
@pc-ctx/core     Types, YAML I/O, CRUD, validation
@pc-ctx/cli      citty commands + Hono server
@pc-ctx/mcp      Import core, expose as tools
```

Core has one runtime dependency: `js-yaml`. The CLI adds citty on top. The MCP server adds nothing. Same parsing, same validation, same types.

This broke initially. I published cli with `workspace:*` in package.json, which pinned core to the exact version. A core bugfix meant republishing cli too. Switched to `workspace:^`. Now minor bumps propagate.

## The embedded web server

`ctx ui --serve` starts a Hono server that serves a React SPA from a local cache and proxies API requests to GitHub. One file, 130 lines.

```typescript
export function startUiServer(port, staticDir, pat) {
  const app = new Hono();
  app.get("/api/:folder", async (c) => {
    /* proxy to GitHub */
  });
  app.get("*", (c) => {
    /* serve static files */
  });
  serve({ fetch: app.fetch, port });
}
```

The SPA isn't bundled with the CLI. It's downloaded from GitHub Releases on first use. 500KB gzipped. Separate repo, separate release cycle.

## Config in a directory

`~/.pc-ctx/config.json` holds one value: a GitHub PAT for the web UI's API proxy. The PAT never reaches the SPA. The server attaches it to API requests server-side.

## No classes

Zero classes. `readAllPlans(dir)` returns `PlanFile[]`. `serializePlanFile(plan)` returns a string. No state, no lifecycle, no dependency injection.

This fits a CLI that reads and writes files. Each command is a pipeline: parse args, read files, mutate, write, print. Classes add ceremony.

## The point

The CLI exists because agents need deterministic access to structured data. MCP is more efficient, but not every agent supports it. The CLI is the fallback. Also what humans use when they don't want a browser.

It doesn't need to be beautiful. It needs to be predictable.
