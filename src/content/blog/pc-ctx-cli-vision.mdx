My brain was the context. I just told the agent what to do every session and hoped I remembered everything it needed. Then I wrote AGENTS.md. Then GEMINI.md when Gemini was still a thing. Then Gemini plugins with skills. Then CLAUDE.md. None of it scaled.

I ended up with a folder that held rules, plans, and session notes so agents could access them without me explaining everything every time. Its structure changed constantly. It started as loose markdown files. Then a hidden `.agent` directory. Then `.context` inside each project (gitignored, so lost on every new clone). Then git-tracked for cross-machine sync. Then prefixed with the workspace name. Then rules leaked into AGENTS.md. Then skills split into separate files. Every few weeks the structure drifted and I told the agent to migrate it. It did.

pc-ctx is what survived after all those iterations. A folder of markdown files in a git repo with an MCP server and a CLI.

## How the data works

Each plan is a markdown file with YAML frontmatter.

```yaml
---
slug: my-plan
title: "My Plan"
status: active
priority: 80
tasks:
  - id: T1
    title: "First task"
    status: done
  - id: T2
    title: "Second task"
    status: pending
---
```

The MCP server exposes plan management as tools. Agents call them natively and get JSON back. The CLI does the same thing, agents without MCP fall back to parsing its output.

The git repo lives on GitHub. Pushes happen on each task or small milestone. Same plans, no sync service.

## Token savings drove this

Before the CLI, agents loaded markdown plan files and skill files listing all the rules. Some plans have a lot of text, and the skill files kept growing. Every session started with the agent reading all of it. Token-heavy and repetitive.

The CLI was the first fix. One command replaced loading a dozen files. Deterministic output, same information, fraction of the tokens.

The MCP server came later and saved even more. Instead of spawning a process and parsing text, agents call functions directly. No parsing overhead on either side. The savings add up when an agent checks status and marks tasks done throughout a session.

Today the MCP server is the primary interface. The CLI is just a fallback.

## What agents actually do with it

I start a session and tell the agent to list what's active. It calls the status tool via MCP. Plans grouped by status, sorted by priority, latest update per project. I pick what to work on and the agent starts. While working, I almost always discover new things. Some become new plans, some go into ideas. Next session, same flow.

Claude Code, Cline, and OpenCode do the same through MCP. Fall back to `ctx status` if MCP isn't available. Same data, three agents, zero custom instructions per agent.

## The cross-machine part

I work across multiple machines. The moment I needed that, I git-tracked the folder. Pushes happen on each task or small milestone. The next machine pulls. The agent calls the status tool and picks up where things stand.

No service, no API, no accounts. Just git.

This is post one in a series about pc-ctx. I want to cover the CLI design, the MCP internals, and the web UI. The core idea is simple: structured data in a git repo, MCP for native agent access, CLI for agents without MCP.
