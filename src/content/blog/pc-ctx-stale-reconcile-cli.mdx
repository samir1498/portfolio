Two MCP tools in pc-ctx v0.4 that handle the bookkeeping I used to ignore. Agents run them. I only reach for the CLI when I want to peek at what happened.

## plan_stale

The agent calls this at session start. It runs three checks against the entire plan directory and returns JSON:

```
all-done:   3 plans where tasks are done but status is still active
idle:       1 plan untouched for 14+ days
focus:      now.md updated 2 days ago
```

**Heuristic 1 -- all-done**: Plans where every task is done but the plan itself is still `active`. Usually the last task completion should have closed the plan but didn't.

**Heuristic 2 -- idle**: Plans where the file modification date is >14 days ago. Could be backlog worth keeping, could be abandoned.

**Heuristic 3 -- focus-stale**: now.md `updated` field is before today. Means I probably haven't set direction for the session.

The agent reads the output and surfaces action items in its summary. I don't need to run `ctx stale` myself. The agent already knows what's stale and tells me.

## plan_reconcile

The agent calls this at session end. It scans git log for `ctx:` trailers and reports what matched:

```
## Matched (3)
  33ad093 2026-06-30 blog-posts-2026 close
  33ad093 2026-06-30 git-commit-convention-skill/T2 progress
  03de051 2026-06-30 standardize-progress-format close

## Unmatched (0)
```

Every matched ref shows the commit hash, date, and action. Unmatched shows what went wrong: plan not found, task ID doesn't exist, file unreadable.

With `--apply`, the tool writes the changes: close marks tasks done with the commit date and sets plan to done if all tasks are resolved. Start sets task to in-progress. Progress moves pending to in-progress.

Roadmap slugs work the same way. Close sets the roadmap status to done and marks all entries done.

## The CLI fallback

The same logic exists as `ctx stale` and `ctx reconcile` in the terminal. I use them when:

- I want to check what the agent reported without parsing its summary
- Something showed as unmatched and I need to investigate
- The agent is not in session and I'm working solo

The output looks the same either way. JSON from the MCP tool, formatted table from the CLI. Same engine, two interfaces.

## Where they sit in the workflow

Session start: the agent calls `plan_stale` to scope the work.

Session end: the agent calls `plan_reconcile --apply` to persist everything.

They don't do much individually. Together they close the loop between "I wrote this in a commit" and "the plan file reflects reality." The fact that agents drive this automatically is the whole point. The CLI exists so I can verify when I want to, not because I need to.
