I wrote the [I Stopped Trusting AI Agents](/blog/stopped-trusting-agents/) post and the [v0.4 release](/blog/pc-ctx-v0-4-release/) post back to back. Both talk about plan_stale and plan_reconcile in abstract terms. This one is what it actually looks like to use them.

Session start. Agent calls `plan_stale`. Output is three buckets:

```
all-done:   3 plans done but still active
idle:       1 plan untouched 14+ days
focus:      now.md updated 2 days ago
```

I scan it in the agent's summary. If the idle plan is something I genuinely abandoned, I don't care. If it's something I forgot about, I pick it. The all-done bucket is the one I actually act on. Close the plan, move on. The focus line is just a reminder to bump now.md.

Session end. Agent calls `plan_reconcile --apply`. This scans the last 50 commits, finds every `ctx:` trailer, cross-references against plan files, and writes the matched ones. Output looks like:

```
Matched (3):   33ad093 blog-posts-2026 close
              03de051 standardize-progress-format close
Unmatched (0)
```

I never read this output unless something is unmatched. That's the part I actually need to see. If a commit says `ctx: broken-slug/T1 close` and the slug doesn't exist, I want to know before the next session starts blind.

The CLI exists. `ctx stale` and `ctx reconcile`. But I use it maybe once every few sessions. Always to investigate unmatched refs. The agent drives the ritual. I just verify the edge cases.

That's the whole thing. Two tools, three lines of output, one thing I check manually.
